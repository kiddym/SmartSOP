# 动态标题字典与自学习方案（L1 显式记忆 + L2 编辑学习闭环）

日期：2026-05-31
作者：架构评审
关联：[解析器准确性评估报告_20260531](解析器准确性评估报告_20260531.md)、`backend/app/parser/`（styles / synonyms / heading_detector）

---

## 0. 一句话目标

把解析器现有的**静态格式字典**升级为**可积累、可自修正的动态字典**：复发模板（同一套样式名/编号体例）**越用越准、人工校核趋近于零**，且全程**可溯源、可回滚、不会被个例污染**。

---

## 1. 现状与接缝盘点（先讲清"能接什么"）

| 现有件 | 位置 | 现状 |
|---|---|---|
| 样式名字典（静态） | `parser/data/heading_synonyms.yaml` | 内置中文同义词，随代码发布，运行时不变 |
| 编号字典 v4（静态） | `heading_detector.py:classify_numbering` | 编号模式→`(kind,level,pattern_key)`，全局代码 |
| 四级反查链 | `styles.py:classify_with_source` | `override → 标准名 → 同义词 → outlineLvl → basedOn`，已返回来源枚举 |
| **注入缝** | `parse_docx(..., style_overrides=)` | **已贯通到结果模型（source="override", conf=1.0），但 `parse_service.py:72` 从不传值 → 恒为 `{}`** |
| 编号模式聚合 | `result.py:DetectedPattern` | 已能按编号模式分组计数，前端可批量提升 |
| 置信度三件套 | `ParsedNode.{confidence_tier, mark_status, heading_source}` | 表达"可信/待核/来源"的现成载体 |
| 审计表 | `models/audit.py`（`ProcedureAuditLog`：action/old_value/new_value）| 可作编辑信号源 |
| 单租户 | `ProcedureSettings` 全局单例 | **无 org/tenant → 字典即全局表**，scope 简单 |

**两个必须补的缺口**：
1. **归因缺口**：`ParsedNode` 与 `ProcedureNode` 都**不带"来源样式名 / 来源编号模式"**——一次"改级"动作无法回溯到是哪个样式/编号被改。这是学习闭环的前提。
2. **接线缺口**：解析时不读任何持久化字典。

## 2. 总体架构

```
                         ┌─────────────── 写回（学习） ───────────────┐
                         │                                            │
   docx ──parse──►  解析链 ──►  审查/导入 ──►  编辑器  ──►  节点改级/正文互转
              ▲          │            │            │
              │      读字典       L1 显式"记住"   L2 隐式信号
        style_overrides  │            │            │
        numbering_over   │            ▼            ▼
              │          │      ┌──────────────────────┐
        ┌─────┴────┐     └─────►│   学习事件表(原始信号)  │  append-only
        │ 动态字典  │◄──聚合/投票──│  + 聚合器 → 规则表      │  可重放
        │ (规则表)  │            └──────────────────────┘
        └──────────┘
```

**两本字典，分轨**（评估报告 §Q2 陷阱③）：
- **样式字典** `tb_heading_style_rule`：覆盖 `heading_synonyms`，按**样式名**。
- **编号体例** `tb_numbering_profile`：覆盖 `classify_numbering` 的 `kind/level`，按**编号 pattern_key**；**仅作用本部署**，绝不改全局代码规则。

**优先级（解析时解析顺序）**：
`手动钉死(manual) > 学习生效(learned active) > 内置同义词/编号规则 > outlineLvl > basedOn`
——即把现有 `classify_with_source` 的 "override" 一级拆成 `manual` 与 `learned`，其余链路不变（**向下兼容、纯增量**）。

## 3. 数据模型（新增 3 张表，全局 scope）

### 3.1 `tb_heading_style_rule` —— 样式字典（解析时读取的"生效规则"）

| 列 | 类型 | 说明 |
|---|---|---|
| id | uuid | |
| style_name | str(255) **uniq** | Word 样式显示名，如「章节标题」|
| level | int·null | 1/2/3；**0/null = 显式判定"非标题/正文"** |
| source | enum | `manual`(显式钉死) / `learned`(自学习) / `disabled` |
| status | enum | `active`(已生效，解析时应用) / `candidate`(证据不足，暂不应用) |
| level_votes | json | 证据直方图 `{"1":23,"2":1,"content":0}`，供一致率与透明展示 |
| evidence_count | int | 去重文档级证据数 |
| agreement | float | `max(votes)/sum(votes)`，一致率 |
| revision | int | 乐观锁（对齐既有 `ProcedureSettings.revision` 习惯）|
| created_at/updated_at/last_evidence_at | dt | |

### 3.2 `tb_numbering_profile` —— 编号体例（按部署覆盖编号判定）

| 列 | 说明 |
|---|---|
| pattern_key str **uniq** | 复用 `classify_numbering` 产出的 `pattern_key`（如 `N.N、` `第X条` `N、`）|
| kind enum | `heading` / `weak_heading` / `list` / `content` |
| level int·null | |
| source/status/level_votes/evidence_count/agreement/revision/时间戳 | 同 3.1 |

> 例：本组织一贯把 `第X条` 当 L3 标题、把 `N.N、` 当 L2 → 各落一条 `learned active`，无需改 `heading_detector.py`。

### 3.3 `tb_heading_learning_event` —— 原始学习信号（append-only，可重放）

| 列 | 说明 |
|---|---|
| id BIGINT autoinc | 仿 `audit.py` 的 `_AUDIT_PK`（MySQL BIGINT / SQLite INTEGER）|
| procedure_id / node_id | 来源 |
| style_name·null / numbering_pattern·null | **归因键**（二者至少其一）|
| signal_type enum | `review_confirm` / `review_override` / `edit_relevel` / `demote_to_content` / `promote_to_heading` |
| from_level·null → to_level·null | 改级前后；`to_level=null` 表示改成正文 |
| confidence_tier_before | 当时档位（判断是"纠错强信号"还是"确认弱信号"）|
| created_at | |

**为何独立于审计表**：原始信号要能在**聚合算法迭代后重新推导规则**（不丢训练数据），审计表语义是"操作留痕"，混用会绑死算法。聚合器是纯函数 `events → rules`，可随时重算。

## 4. 解析链接通（M1，低风险、解锁一切）

### 4.1 归因字段下穿（补缺口①）

- `ParsedNode` 增 `source_style_name: str|None`、`source_numbering_pattern: str|None`。
  - structurer 命中样式标题时，由 `block.style_id` 经 `StyleIndex` 反查样式名填入；
  - 命中编号时，填 `classify_numbering(...).pattern_key`。
- `ProcedureNode`（持久层）同样**新增这两列**（alembic 迁移，nullable，导入时写入）——
  这样**几周后**用户在编辑器改级，仍能回溯到来源样式/编号（L2 学习的硬前提）。

### 4.2 读字典 → 注入

`parse_service.parse()` 内：解析前从 `tb_heading_style_rule(status=active)` 拼出 `style_overrides: {style_name: level}`，从 `tb_numbering_profile(status=active)` 拼出 `numbering_overrides`，传入：

```python
result = parse_docx(data, mode, style_overrides=style_map, numbering_overrides=num_map)
```

- `style_overrides`：现成参数，`classify_with_source` 已支持（仅需把来源细分 manual/learned）。
- `numbering_overrides`：`classify_numbering` 增一个查表前置——命中 profile 直接返回覆盖后的 `(kind,level)`，否则走内置规则。**level=0/content 时显式当正文**（支持"这个编号不是标题"）。

> M1 完成后，即使没有任何学习算法，管理员也能在字典管理页**手动**维护规则，解析立即生效——这本身就是 L1 的最小可用形态。

## 5. 学习闭环（L1 显式 + L2 隐式）

### 5.1 信号采集点

| 触发 | signal_type | 强度 |
|---|---|---|
| 审查页对 review 项点「确认为 Lx」 | `review_confirm` | 中（人工确认）|
| 审查页把标题改级 / 改正文 | `review_override` | 强（人工纠错）|
| 编辑器改 `heading_level` | `edit_relevel` | 强 |
| 编辑器 标题→正文 | `demote_to_content` | 强 |
| 编辑器 正文→标题 | `promote_to_heading` | 强 |

采集实现：在**节点更新服务**里，当 `heading_level` 或 content/heading 形态变化、且该节点带来源归因时，写一条 `learning_event`（与现有保存逻辑同事务，无额外前端摩擦）。审查页的确认/改级同理。

### 5.2 聚合 → 规则（投票，绝不 n=1）

聚合器对每个 `style_name`（或 `pattern_key`）统计 `level_votes`，按阈值决策：

```
evidence_count ≥ K(默认3, 去重文档)  且  agreement ≥ A(默认0.8)
    → status=active（解析时应用，免 review）
否则
    → status=candidate（不应用，维持现状 review）
冲突(agreement<A)
    → 保持 candidate + 标记 conflict，交管理员裁决，永不自动应用
```

### 5.3 归因粒度护栏（评估报告 §Q2 陷阱①，最危险）

**段落级个例 ≠ 样式级真理。** 直方图天然挡住：TP试验程序把 1 句滥用 `章节标题` 的正文降级 → `{"1":23,"content":1}`，agreement=0.96 仍判 L1，**不翻规则**。只有当**多数**同样式节点被改成同一目标，规则才动。这把"样式被滥用"与"样式被一致纠错"自动分开。

## 6. 防误学护栏（回答 Q1：还要不要人工校核）

校核成本从「每文档每标题」**摊薄**为「每个新样式一次 + 异常抽检」，分状态：

| 样式状态 | 前端是否人工 | 机制 |
|---|---|---|
| 未知/candidate | **要**（review，照旧）| 这正是字典录入来源 |
| learned active | **免逐条**，但保留**异常哨兵+抽检** | 见下 |
| manual 钉死 | 免 | 管理员权威 |

**即使 active 也保留的内容级哨兵**（命中即从 HIGH 降回 review，不丢、不改字典）：
- 样式滥用：`_looks_like_prose`（现有）——整句正文套了可信样式；
- 层级跳变：L1 直接跳 L3、同级编号断裂；
- 异常长标题 / 标题含完整句读。

**防自锁**：active 规则若开始累积**矛盾**编辑（agreement 跌破 A'），聚合器自动降级为 candidate + 告警——避免"学错了→自动接受→没人再纠→永远错"。

**可溯源 / 可回滚**：每条规则展示 `source + 证据数 + 直方图`（"判 小标题=L2 因 7 文档 23 标题确认"），管理员可钉死 / 禁用 / 回滚 / 导出导入。

---

## 7. 前端交互（三处）

1. **审查页**：每个 review 标题行加「记住「样式名」= L1/L2/L3」「此样式非标题」→ 写 `manual` 规则（即时 active）+ 一条 `review_override` 事件。（L1）
2. **编辑器**：改级/互转**静默**发学习事件，无摩擦；规则由 candidate 升 active 时弹一次轻提示「已学习：样式X 默认识别为 L2（基于 N 次确认）」，给透明度。（L2）
3. **字典管理页**（设置内）：列出 内置/learned/manual 三类规则、证据、冲突；启用/禁用/回滚/钉死/导出导入；编号体例同页另一 tab。

**关键实现要点（投票口径，落实三道护栏）**：
- **哪些文档算证据**：仅「该样式在该文档有学习事件（用户校验过）」的文档计入——杜绝把解析器自身输出当证据的循环自证。
- **每文档投 1 票 = 该文档内该样式 active 节点的主导最终层级**（取**节点终态**，非事件流）。这是归因粒度护栏的实现：TP试验程序 23 个 L1 标题 + 1 句滥用降级 → 该文档主导 L1、只投 1 票 L1，单句例外不单独成票、`章节标题=L1` 不被翻动（已单测固化）。
- **手动优先**：`source='manual'`（「记住此样式」）规则不被自学习覆盖。
- 生效（active）规则立即进入解析注入（`active_style_overrides`），形成「越用越准」闭环（已单测：3 文档一致改级 → 自动 active → 注入字典）。

**M3 已知边界（明确延后）**：
- **冷启动**：完全未被任何信号识别的样式（解析后为正文、节点无 `source_style_name`）无法隐式学习——需在 content 节点也带归因，属后续增强。当前隐式学习覆盖「已识别为标题但层级需纠正」这一高频真实场景。
- **「强制非标题」**：多数被降为正文的样式只记 candidate、不在解析中强制压制（需 parser 增 `force_content` 通路），延后。
- **升级提示 toast**（§7.2）未做；规则升 active 当前静默（可在 M4 字典管理页查看）。

## 9. 验收与回归（用现有 36 份样本闭环验证）

- **回归门**：标准样式 4 份 + TP试验程序 解析层级**不得回归**（沿用本轮 P0/P1 修复后的基线）。

- **防污染用例**：TP 滥用样式的单句降级**不得**翻动 `章节标题=L1` 规则（agreement 守住）。
- **冲突用例**：构造同样式名在两套模板里分别为 L1/L2 → 规则保持 candidate + conflict，不自动应用。

## 10. 关键结论

- **能积累、能自修正**：架构接缝（`style_overrides`/来源枚举/置信度三件套/审计）已就位，L1+L2 是**纯增量**，不动解析主干。
- **唯一硬前提**是补"来源归因"字段（M1），否则学习无法把"改级"绑回"哪个样式"。
- **人工校核不会归零、但会收敛**：复发模板趋近零，异常驱动抽检长存——这正是"既省人力又不失控"的正确平衡。
- **三道护栏缺一不可**：投票阈值（防 n=1）、归因粒度（防个例污染）、异常哨兵+矛盾降级（防自锁）。
