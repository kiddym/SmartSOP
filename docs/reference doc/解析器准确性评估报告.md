# Word 解析器准确性评估报告

测试日期：2026-05-30
测试范围：`docs/reference doc/typical word doc/` 下全部 35 份 .docx（根目录 6 份 + `extra doc/` 26 份 + `无格式标题word/` 5 份）
被测组件：`backend/app/parser/`（三阶段管线 Normalizer → Structurer → 序列化），`standard` 与 `smart` 双模式
方法：对每份文档抽取"原始标题真值"（样式名含 heading/标题、或 pPr/outlineLvl、或 numPr 多级编号），与解析输出章节树逐项对照；并核对 metadata、warnings、置信度分级。PDF 预览（`PdfPreviewDialog.vue`）作为层级核对的可视化辅助。

---

## 一、总体结论

| 文档类型 | 样本 | standard | smart | 标题召回 | 层级准确性 |
|---|---|---|---|---|---|
| 标准样式标题（heading/标题 N） | 公司运营管理、公司运营管理_表格图片、电厂管理巡视规定、1_程序模板 | ✅ 通过 | ✅ 通过 | ~100% | ✅ 准确 |
| 单一自定义样式 + Word 多级编号 | TP试验程序 | ✅ 通过但**层级压平** | 同 standard | ~100% | ❌ **三级压成一级** |
| 零样式 + 纯数字/中文编号 | extra doc 26 份、无格式标题 5 份 | ⛔ 正确拒绝（模板校验 error） | ✅ 启发式恢复 | 高 | ✅ 编号层级恢复准确 |
| 目录/索引型文档 | 程序文件目录 | ⛔ 拒绝 | 空树（合理） | — | — |

**核心判断**：解析器在标准样式文档与零样式编号文档上表现良好；存在 **2 个确定缺陷**（1 个影响层级正确性、1 个完整性校验误报）和若干设计权衡点。

## 二、关键发现

### 🔴 P1（严重）标题层级被压平 —— 忽略段落级 `numPr/ilvl` 与 `outlineLvl`

**现象**：`TP试验程序.docx` 原文有清晰三级结构（如 `准备 > 工作票准备/实验室准备`、`现场试验 > 检查隔离边界/安装设备`），解析结果却是 **43 个并列 H1**，层级全部丢失。

**证据**：该文档 64 个标题段落 **全部** 使用同一自定义样式 `章节标题`，真实层级编码在 Word 多级自动编号里：

```
numPr 层级分布：numId=1,ilvl=1 → 28 段（应为 L2）
                numId=1,ilvl=2 → 11 段（应为 L3）
                numId=2/6/11,ilvl=0 → 顶层
样式分布：{'章节标题': 64}   ← 单一样式名
```

**根因**：`styles.classify_with_source()` 推导标题层级时，信号优先级为
`style_overrides → 标准名 → 同义词 → 样式自身 outlineLvl → basedOn 链`。
`章节标题` 命中同义词词典固定映射到 **level 1**（`heading_synonyms.yaml:11`），而 `structurer._classify_heading()` 对样式标题直接返回 `block.style_level`，**从不读取段落级的 `numPr/ilvl`，也不读取段落级 `outlineLvl`（`block.outline_lvl` 被采集但全程未使用）**。一个样式名只能映射一个层级，于是所有标题塌陷到同一级。

`heading_synonyms.yaml:8` 注释已确认此为已知行为：「TP试验程序.docx 的 65 个标题全靠『章节标题』命中」——开发知晓全部落到 level 1，但未补层级细化。

**影响**：所有"单一标题样式 + 依赖 Word 多级列表/大纲级别表达层级"的文档（中文核电规程、国标 SOP 中极常见）层级全失。在 PDF 预览中表现为目录（TOC）43 条无缩进的并列条目，与原文三级大纲明显不符。

**修复方向**：
- 在 Normalizer 采集段落 `numPr` 的 `numId+ilvl`（目前只存了布尔 `numbered`）。
- Structurer 对样式标题判级时，引入"段落级细化"：当**同一扁平样式**下的标题携带 `ilvl`/`outlineLvl` 差异时，用其相对深度细化层级（`ilvl` 为 0-based，`ilvl+1` 映射 L1/L2/L3）。
- 注意 `numId` 命名空间隔离（TP 中正文大纲是 `numId=1`，插入的列表是 `numId=6/11`），跨 `numId` 的 ilvl 不可直接比较，应以"文档主大纲 numId"为基准或回退到段落 `outlineLvl`。

### 🟠 P2（中）段落完整性校验 C003 系统性误报

**现象**：几乎每份含表格的文档（**包括参考模板 `1_程序模板.docx` 自身**）都触发 `段落可能遗漏` warning：

| 文档 | 报告"保留率" | 实际 |
|---|---|---|
| 1_程序模板（参考模板） | 50.6% | 无丢失 |
| TP试验程序 | 21.7% | 无丢失 |
| 电厂管理巡视规定 | 24.2% | 无丢失 |
| 公司运营管理 | 43.1% | 无丢失 |

**根因**：`completeness.paragraph_count_match()` 的分母 `nd.raw_paragraph_count = body.iter(w:p)` 统计了 **表格单元格内、文本框内、以及空段落** 的全部 `<w:p>`；而分子 `kept` 只数 **顶层 paragraph block**。单元格段落被正常折叠进 table 块的 HTML（并未丢失），文本框段落被 hoist 成独立块，于是比值被严重低估。

实测分解（`公司运营管理.docx`）：`body.iter(w:p)=195` = 顶层非空 33 + 顶层空 51 + **表格内 111**；`kept=84`。111 个单元格段落全部保留在 table HTML 中，并非丢失。

`completeness.py:36` 的 docstring 断言「当前 normalize 与 raw 严格 1:1，预期总是 pass」与事实不符。

**影响**：用户在解析审查界面看到的"段落可能遗漏"提示在绝大多数真实文档上是假警报，会侵蚀用户对完整性提示的信任（狼来了效应）。

**修复方向**：分母改为"body 顶层 `<w:p>`（排除 `tc`/`txbxContent` 子树内的段落）"，与分子口径对齐；或对单元格段落单独对账（已有 `raw_image_count`/`raw_table_count` 的同口径模式可借鉴）。

### 🟡 P3（低 / 设计权衡）零样式文档全部标 `review`，确认负担重

**现象**：零样式编号文档（如 `25-各级人员…` 50 章、`02记录控制程序` 22 章）启发式命中**全部** `mark_status=review`，其中 LOW 档占多数（25 号 45 个 LOW，02 号 21 个 LOW）。

**评估**：这是 `§25.5 启发式封顶 0.84、永不自动 HIGH` 的有意设计——非标准标题必经人工确认。实测**精度其实很高**：`02记录控制程序` 恢复出 `1/2/3` 一级、`3.1–3.4` 二级、`4.1.1/4.1.2` 三级，层级完全正确；`25-各级人员` 正确恢复 `4.x 角色 > 4.x.1 职责/权限`。即"多判"指标（真值=0 时一切命中都算多判）是失真的，并非误报。

**权衡建议**：可考虑对**高置信编号模式**（如全文统一的 `N.` / `N.N` 且字号一致）整组提升档位或默认勾选，降低用户逐条确认成本（`detect_patterns` 已具备按组聚合能力，前端可做"按模式批量确认"）。

## 三、其他观察（非缺陷）

1. **standard 模式对零样式文档正确拒绝**：26+5 份零样式文档全部触发 `PARSE_TEMPLATE_INVALID`（模板校验 error），符合"标准模式只信样式、违规即拒"的设计，引导用户改用 smart。
2. **作者样式误用，解析器忠实反映**：`TP试验程序` 中整句话（如"本试验中使用的气体放射性碘131，需建立临时控制区"）被作者套用了 `章节标题` 样式，解析器忠实当作章节。属"garbage in"，解析器无法也不应自动甄别，但可作为前端审查提示（标题异常长 → 建议复核）。
3. **图片/表格抽取正确**：`公司运营管理_表格图片`（img=1 tbl=1）、`TP试验程序`（img=2 tbl=4）、`电厂管理巡视规定`（img=1 tbl=3）图表数与原文一致，C001/C002 对账通过。
4. **目录/字段/页眉页脚处理稳健**：TOC 字段域、SDT、文本框、vMerge 表格、页眉页脚丢弃告知均正常工作。

## 四、PDF 预览作为辅助核对的价值

`PdfPreviewDialog.vue` 渲染的是**导入后的程序树**（A4 分页 + 目录 TOC + `h1/h2/h3` 缩进 + 章节编号 code），是检验解析层级正确性的最佳可视化手段：

- **目录（TOC）缩进**（`.toc-l1/l2/l3`）直接暴露 P1 层级压平：TP试验程序的 PDF 目录会显示 43 条无缩进并列项，与原 Word 三级大纲一眼可辨差异。
- **章节编号 code**（`node_numbering` 重算）可核对编号连续性与层级深度。
- **续页标题 `（续）`** 与分页边界红线辅助核对内容归属是否正确。

建议在解析审查流程中，将"PDF 预览 TOC vs 原文目录"作为层级正确性的标准核对步骤；P1 修复后应以此回归验证 TP 类文档。

## 五、优化建议优先级汇总

| 优先级 | 问题 | 建议 |
|---|---|---|
| P1 | 单一样式 + 多级编号层级压平 | Normalizer 采集 `numPr(numId,ilvl)`；Structurer 用段落级 `ilvl`/`outlineLvl` 细化扁平样式标题层级 |
| P2 | C003 完整性校验误报 | 分母排除单元格/文本框/空段落，与分子顶层段落口径对齐 |
| P3 | 零样式文档确认负担重 | 对统一高置信编号模式支持"按组批量确认"/默认勾选 |
| 观察 | 标题样式误用 | 前端对异常长标题给出复核提示 |
