## 一、总体结论

| 文档类型 | 样本 | standard | smart | 标题召回 | 层级准确性 |
|---|---|---|---|---|---|
| 标准样式标题（heading/标题 N） | 公司运营管理、_表格图片、电厂管理巡视规定、1_程序模板 | ✅ 通过 | ✅ 通过 | ~100% | ✅ 准确 |
| 单一扁平样式 + Word 多级编号 | TP试验程序 | ⚠ 通过但层级误判 | 同 standard | ~95% | ❌ **跨 numId 层级/嵌套错误** |
| 零样式 + 顿号编号（`5.1、xx`） | extra doc 多数（12/17/18/22…） | ⛔ 正确拒绝 | ⚠ **仅顶层召回，子节全丢** | L1 高 / L2-L3 低 | ⚠ 子节层级缺失 |
| 零样式 + 空格/点/直连编号（`4.1 xx`） | 25-各级人员、02记录、05人力 | ⛔ 正确拒绝 | ✅ 启发式恢复 | 高 | ✅ 层级准确 |
| 目录/索引型 | 程序文件目录 | ⛔ 拒绝 | 空树（合理） | — | — |

**核心判断**：标准样式文档表现优秀；零样式文档的召回**强烈依赖编号分隔符**——`N.N 空格/点/直连中文` 子节召回良好，但 **`N.N、`（中文顿号）子节几乎全部丢失**，这是本次最重要的准确性缺陷（中文 ISO 程序文件顿号写法极普遍）。

## 二、关键发现

### 🔴 P0（新，严重）顿号编号子节标题被整组丢弃 —— 影响 extra doc 多数文档

**现象**：`12-产品和服务的要求控制程序.docx` 原文有清晰两级结构（`5、工作流程` 下含 `5.1、顾客沟通`、`5.2、顾客要求确认`、`5.3、合同评审`、`5.5、服务`；`7、表单` 下含 `7.1、订货合同` 等），但 smart 解析结果是 **8 个并列 L1，零个子节**：

```
PDF 预览 TOC（模拟）：
产品和服务的要求控制程序 / 1、目的 / 2、范围 / 3、权责 / 4、定义 / 5、工作流程： / 6、相关文件 / 7、 表单
（5.1–5.5、7.1–7.3 全部沦为 content，TOC 中不可见）
```

**证据（评分实测）**：

| 段落 | bold | 分隔符 | classify_numbering | score | tier | 结果 |
|---|---|---|---|---|---|---|
| `1、目的` | **1.00** | 顿号 | weak_heading | 0.55 | medium | ✅ 升 |
| `5.1、顾客沟通` | 0.00 | 顿号 | **weak_heading** | **0.10** | None | ❌ 丢 |
| `4.1 总经理`（25号） | 0.00 | 空格 | **heading** | 0.35 | low | ✅ 升 |

**根因**：双重惩罚，均落在 `heading_detector.py`：
1. **编号分类**（`classify_numbering` `heading_detector.py:86-88`）：`N.N、`（顿号）一律判 `weak_heading`，而 `N.N 空格`（:96）、`N.N.` 点（:90）、`N.N直连中文`（:100）都判 `heading`。注释「顿号歧义 weak（Q217）」的本意是抑制 depth=1 的正文顿号条款（如危险源 `1、设有消防…`），但**对 depth≥2 的点分编号一并误伤**——`5.1、顾客沟通` 这种「点分前缀 + 短名词标题」其实毫无歧义。
2. **档位提升门槛**（`structurer._classify_heading` `:268-273`）：LOW 档仅在 `num.kind=="heading"` 时才提升；`weak_heading` 被排除。叠加子节通常**不加粗**（作者只给顶层 `1、` 加粗），`weak_heading + 非粗 + 短` 评分仅 0.10，连 LOW 阈值（0.3）都够不到。

**对比印证**：`25-各级人员`（`4.1 总经理` 空格分隔）召回 100%（`{1:5,2:15,3:30}`）；`02记录`/`05人力`（空格/点分隔）召回 100%+。而 `12/17/18/22 号`（顿号子节）只剩 `{1:8}` 一层。**唯一差异就是分隔符**。

**影响**：中文 QMS/ISO 程序文件中 `5.1、` 顿号写法是主流，本缺陷使这类文档在 smart 模式下**丢失全部二、三级结构**，PDF 预览 TOC 退化为一层平铺，与原文目录明显不符。

**修复方向**：放宽 `classify_numbering`——`depth≥2` 的 `N.N、`/`N.N.N、` 判 `heading`（点分前缀已表达层级，顿号在此非歧义）；仅 `depth==1` 的 `N、` 保留 `weak_heading`（维持危险源正文条款抑制）。此改动可一次性恢复 12/17/18/22 等多份文档的子节层级。

### 🔴 P1（残留，严重）扁平样式跨 numId 层级细化错误 —— TP试验程序

**现象**：`TP试验程序.docx` 旧报告的「43 条并列 L1」已部分改善（现 `{1:32, 2:11}`），但层级仍**系统性错位**：

```
PDF 预览 TOC（模拟，节选）：
准备
  · 工作票准备        ← L2 ✓
  · 实验室准备        ← L2 ✓
  · 文件准备          ← L2 ✓
碘样品分析单已填写、待用          ← ❌ 应为「文件准备」下的 L3，却升成 L1
从隔离办领取使用外源工作许可证     ← ❌ 同上
向运行人员了解机组状态…           ← ❌ 同上
  · 现场准备          ← ❌ L2，但因前面错位被错挂到「向运行人员了解」之下
```

**证据（numPr 实测）**：

| 标题 | numId | ilvl | 应有层级 | 实际 |
|---|---|---|---|---|
| 准备 | 1 | 1 | L1 | L1 ✓ |
| 工作票准备/文件准备/现场准备 | 1 | 2 | L2 | L2 ✓ |
| 碘样品分析单已填写… | **11** | **0** | L3（文件准备下） | **L1** ❌ |

**根因**：`structurer._refine_styled_level`（`:204-221`）按 **每个 numId 独立** 取最小 ilvl 作归一基线（`num_floor`）。`碘样品分析单` 等清单是作者新插入的独立列表 `numId=11`，其 `ilvl=0` 经归一后 → 相对深度 0 → L1，但它实际嵌套在 `文件准备`(L2) 之下应为 L3。由于跨 numId 的 ilvl 不可直接比较、且该归一无法感知列表在文档层级中的真实位置，独立子列表被整体抬到顶层；继而 stack 逻辑使其后的 `现场准备`(L2) 错挂到错误父节点。

> 这正是旧报告「修复方向」已预警的点：「跨 numId 的 ilvl 不可直接比较，应以文档主大纲 numId 为基准或回退到段落 outlineLvl」。当前实现用 per-numId min-floor 归一，未解决跨 numId 嵌套。

**修复方向**：
- 优先用段落 `outlineLvl`（最权威）；TP 这批段落 `outlineLvl=None`，故需更稳的兜底。
- 对扁平样式标题，引入「**按文档顺序的相对缩进栈**」而非 per-numId 归一：遇到新 numId 时，其深度应**继承当前所在父级深度 + 1**，而不是重置为 1。
- 或：仅当全文档**单一 numId** 时启用 ilvl 细化（TP 的主大纲是 numId=1）；多 numId 混排时回退到不细化（避免错误抬层），把这类标为 review 提示人工复核。

### 🟠 P2（已修复，记录）C003 段落完整性误报已解决 ✅

旧报告 P2 指出几乎每份含表文档都误报「段落可能遗漏」。本次复测**所有 36 份文档均未再触发** completeness warning。`completeness.paragraph_count_match`（`:29-43`）分母已改为 normalize 阶段的过滤计数（顶层 + 文本框段落，排除单元格直属段落），与分子口径对齐。该缺陷已闭环。

### 🟡 P3（设计权衡）零样式文档全部标 review

零样式文档启发式命中全部 `mark_status=review`（§25.5 封顶 0.84、永不自动 HIGH 的有意设计）。在召回正确的文档（25号、02记录、05人力）上层级精度很高，确认负担是可接受的设计取舍。但**与 P0 叠加时**问题被放大：顿号文档不仅要逐条确认，连子节本身都没被识别出来，用户无从「确认」一个不存在的节点。**修复 P0 后**再考虑「按高置信编号模式批量确认」以降低勾选成本。

## 三、其他观察（非缺陷）

1. **standard 模式对零样式文档正确拒绝**：extra doc / 无格式 全部触发 `H001`（章节数=0、validation 不通过），符合「标准模式只信样式」设计，引导改用 smart。
2. **作者样式误用，解析器忠实反映并提示**：TP试验程序中整句话（`本试验中使用的气体放射性碘131，需建立临时控制区`）被套了 `章节标题` 样式，解析器忠实当章节但已正确标 `review`（`_looks_like_prose` 生效）；惟其后几句更短的误用句（`通知辐射防护人员`）因不触发句读/长度阈值未标 review。
3. **图片/表格抽取正确**：`公司运营管理_表格图片`(img=1,tbl=1)、`TP试验程序`(img=2,tbl=4)、`电厂管理巡视规定`(img=1,tbl=3) 与原文一致，C001/C002 对账通过。
4. **页眉/页脚/批注丢弃告知正常**：`discarded_by_design` warning 在含 header/footer/comments 的文档上稳定产出。
5. **`无格式标题word/有限空间作业管理办法`**：真值正则未匹配到点分编号（该文用「第X条」体例），smart 仍恢复 7 个 L1，属正常。

## 四、PDF 预览作为辅助核对的价值

`PdfPreviewDialog.vue` 渲染导入后的程序树（A4 分页 + 目录 TOC + `h1/h2/h3` 缩进 + 章节编号），其 **TOC 缩进（`toc-l1/l2/l3`，按 `node.level`）** 是检验层级正确性的最直观手段：

- **P0 暴露**：`12号` 的 PDF TOC 只有一层 8 条，原文 `5.1/5.2/7.1…` 子节全部不可见——与原 Word 目录一眼可辨差异。
- **P1 暴露**：`TP试验程序` 的 PDF TOC 中 `碘样品分析单…` 与 `准备` 同级并列、`现场准备` 缩进位置异常——直接显出跨 numId 嵌套错误。

建议把「PDF 预览 TOC ↔ 原文 Word 目录」作为解析审查流程的标准核对步骤；P0/P1 修复后以本文两份文档做回归。

## 五、优化建议优先级汇总

| 优先级 | 问题 | 建议 | 落点 | 状态 |
|---|---|---|---|---|
| **P0** | 顿号子节 `N.N、` 整组丢失 | `depth≥2` 顿号判 `heading`（仅 `depth==1` 保留 weak） | `heading_detector.py:86-88` | ✅ 已修复 |
| **P1** | 扁平样式跨 numId 层级/嵌套错误 | 文档顺序相对嵌套（`_assign_styled_depths`），扁平样式才启用 | `structurer.py` | ✅ 已修复 |
| P3 | 零样式确认负担 | 修复 P0 后支持「按高置信编号模式批量确认」 | 前端 + `detect_patterns` | ⏳ 后续 |
| 观察 | 短误用句未标 review | `_looks_like_prose` 增补「与上下文字号/编号不一致」信号 | `structurer.py` | ⏳ 后续 |

**结论**：解析器在标准样式与「空格/点分编号」零样式文档上准确可靠；当前两个确定缺陷（**P0 顿号子节丢失**、**P1 跨 numId 层级错位**）集中影响中文 SOP 的二、三级结构还原，建议优先修复 P0（影响面最广、改动最小、收益最大）。

## 六、修复执行记录（2026-05-31）

### P0 修复 —— 顿号子节召回

**改动**：`heading_detector.py` `classify_numbering`，`N.N、`（`depth≥2`）改判 `heading`（与 `N.N 空格`/`N.N.`/`N.N直连中文` 同等），仅 `N、`（`depth==1`）保留 `weak_heading`（维持危险源 `1、设有消防…` 正文条款抑制不回归）。

```python
if rest.startswith("、"):
    if depth >= 2:
        return NumberingMatch(kind="heading", level=level, pattern_key="N" + ".N" * (depth - 1) + "、")
    return NumberingMatch(kind="weak_heading", level=level, pattern_key="N、")
```

**验证**：`5.1、顾客沟通` 评分 0.10→0.45（达 LOW，kind=heading）→ 正确升为 chapter。

### P1 修复 —— 扁平样式跨 numId 嵌套

**改动**：`structurer.py` 新增 `_assign_styled_depths` 预处理（在主循环前按文档顺序计算每个样式标题深度）。**仅对扁平样式文档**（全部样式标题同一 `style_level`）启用：把「非主大纲 numId 且出现在子节（深度≥L2）之内」的次要子列表嵌套为当前 section 的下一级，而非被 per-numId 归一抬回 L1。样式已编码层级的规范文档（标题1/2/3）走基线、零改动、无回归。

**验证（TP试验程序）**：

```
准备
  · 工作票准备 / 实验室准备 / 文件准备
      ‣ 碘样品分析单已填写、待用        ← 修复前 L1，修复后正确 L3（文件准备下）
      ‣ 从隔离办领取使用外源工作许可证
      ‣ 向运行人员了解机组状态…
  · 现场准备                          ← 修复前错挂，修复后正确回到「准备」下 L2
```
