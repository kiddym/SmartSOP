# 统一创建/编辑模型 · 总览设计（北极星）

**日期：** 2026-05-25
**状态：** 已批准（总览层）；各 Phase 细节待各自 spec
**作者：** 协作设计（cui_yuming + Claude）
**性质：** 北极星总览 spec。固化模型与核心决定、划定子项目边界。**不含**实现级细节——那些进各 Phase 自己的 spec。

---

## 1. 背景与问题

当前从 Word 创建程序要走一条独立的 **V2 beta 导入对话框**（`ImportDialog.vue` 等 `import-v2/*`）：上传 → 解析 → 在对话框里做结构分检（层级标定、步骤标注、忽略、改标题）→「提交导入」落库 → 跳转到程序编辑器继续编辑。此外还存在一条**老导入路径**。

痛点：

- **两套编辑界面**：导入对话框的右侧详情是**只读预览**（正文 `rich_content` 不可改、步骤不可编），真正的编辑要等导入后在程序编辑器里做。同一件事维护在两处，易发散。
- **原文只在导入期可见**：编辑器里没有 Word 原文预览；一旦导入，对照源文件就难了。而解析结构**常大面积出错**，纠错严重依赖对照原文。
- **创建路径分裂**：beta2 导入、老导入、空白新建是三条不同的入口/心智。

## 2. 目标与非目标

**目标**

- 把"导入"从一个独立目的地，降级为"新建程序时的一个**播种**选项"。
- **一个编辑器**承载所有创建/编辑；Word 原文预览作为可折叠栏常驻可用、且**长期可追溯**。
- 层级标定/步骤标注能在编辑器里**随时重做**，且**绝不静默丢失**已有编辑。
- 下线 beta2 导入对话框与老导入路径。

**非目标（本轮不做）**

- 重写程序编辑器的核心编辑能力（富文本、表单字段、附件）——复用现有 `RichTextEditor`、`StepFormFields` 等。
- 解析算法（`/parse`）本身的改造。
- 多人协同/实时编辑。

## 3. 统一模型（北极星）

**一个编辑器，两种出生方式，"导入"= 播种：**

```
新建程序 ┬─ 空白           → 进编辑器（空树，无预览栏）
         └─ 从 Word 导入   → 上传 → 解析 → 建 draft 程序 → 进编辑器
                              （可折叠 Word 预览栏 + 活树已播种
                               + 5 选项批量层级标定 + 待确认 triage）
                                        │
                                  编辑 / 重标 / 标步骤（随时）
                                        │
                                  点「完成」→ 翻正式（校验 / 清待确认 / 定稿编号）
```

`beta2 导入对话框`、`老导入路径`、`import-v2/*` 的只读详情卡片**全部下线**。

### 关键简化（务必理解）

因为选择了**上传即建 draft**（见决定 2），**编辑器始终面对一个已落库的程序（draft 或正式）**，永远不需要"内存解析树"这种第二数据模型。`/parse` 的结果在**播种那一刻**由后端一次性翻译成正常的程序（复用现有 `ImportNode 树 → chapters+steps` 的 import 逻辑）。因此：

- 不存在"两套数据模型要同步"的难题——只有**一棵活的、已存库的程序树**。
- 编辑器的加载路径基本沿用现有 `store.load(id)`，主要是**增加 draft/待确认/置信度的感知**，而非新增一种加载模式。

## 4. 已锁定的核心决定

### 决定 1 · 层级标定 = 活树上原地重排

- 层级标定作用对象 = **编辑器里这棵已存库的活树**（不是一次性导入树）。
- 操作：把当前树**拍平成文档顺序**，逐行给 5 选项角色按钮；应用时用**现有节点身份原地重新挂父子/层级**——**只移动，绝不推倒重建**。
- 因此**内容天然不丢**：节点（连同正文编辑、步骤表单字段）只被移动，从不被销毁重建。
- **新增/拆分/合并**的节点作为普通行参与重排，无需"原始段落层"、无需锚定记账。
- 算法可复用 beta2 `buildTreeFromRoles` 的"复用节点数据重排"思路，输入换成编辑器节点类型。
- **不静默丢失原则**："重标永不静默丢东西"。任何无法干净归位的，进可见的"未归位/待处理"出口，由用户处置，绝不无声丢弃。

### 决定 2 · 草稿生命周期 = 上传即 draft +「完成」里程碑

- 上传/解析后**立刻建 `draft` 状态的程序**，随即进编辑器。
- 编辑/重标随意；点**「完成」**才翻成正式状态，顺带：跑校验、要求清空（或确认接受）"待确认"、定稿编号。
- draft 在程序列表**单独分组/可过滤**；长期未完成的 draft 可**自动清理**。
- "确认前不落库"软化为"**确认前是 draft**"——安全感保留，列表保持干净。

### 决定 3 · 原始 docx 永久存储、可追溯

- 原始 docx 作为程序的 **asset 永久存储**，按程序 id 取回渲染。
- Word 预览栏在 **draft 与正式状态都随时可看**；正式程序**可长期追溯回源 Word**（受控文件场景的硬需求）。

### 决定 4 · 层级 + 步骤合并为 5 选项批量视图

- 一个批量视图同时定层级与步骤类型：**一级 / 二级 / 三级 / 正文 / 步骤**，逐行选、原地重排。
- 把一行标成"步骤"即转步骤类型；**降型转换藏存不丢**——转走时把目标类型放不下的类型专属数据（步骤的 `input_schema` / `expected_output` / 附件标记等）**藏存在节点上**（不显示、不删除），日后转回即恢复；彻底删除须走编辑器的显式删除。

### 已确认的两个配套项

**待确认 / 置信度 triage 的落点**：把 beta2 顶部那套搬进编辑器——头部显示「N 个待确认」；树行带"待确认"徽标；提供"只看待确认"过滤；逐节点"接受"；**点「完成」时若仍有待确认则拦截/警告**。

**统一入口**：程序列表「新建」拆成「空白新建」「从 Word 导入」；两者都建 draft 并进编辑器；Word 那条带可折叠预览栏（复用已落地的可折叠面板能力，见 `2026-05-25-collapsible-import-panels-design.md`）。

## 5. 数据模型蕴含（原则层，细节进各 Phase）

- **单一活树**：程序树（chapters / 正文 / steps）是内容唯一真源；层级标定原地重排它。
- **节点新增字段**（具体落位进 P1/P2 spec）：
  - 待确认/置信度：从解析结果带入（`mark_status='review'`、`confidence_tier` 等）。
  - 降型藏存：保存被降型时丢弃的类型专属数据的隐藏槽。
- **程序状态**：`ProcedureMeta.status` 增加（或复用）`draft` 态及 draft→正式的迁移。
- **原始 docx**：作为程序 asset 存储（复用现有 `/procedures/{id}/assets` 机制）+ 一个取回/渲染入口。
- **播种**：`/parse` 结果在建 draft 时由后端翻译成程序树（复用现有 import 翻译逻辑）。

## 6. 分阶段拆分（路线图）

横跨前端 + 后端 + 存储 + 迁移，拆成三个子项目，**各自 spec → 计划 → 实现**。每个都应产出可独立测试、可工作的软件。

- **P1 · 后端基座**
  draft 生命周期（上传→draft、「完成」finalize、`status`、列表过滤、废弃清理）；原始 docx 永久存储与取回端点；把现有 import 翻译逻辑改造成"播种 draft"。
- **P2 · 编辑器内核**
  编辑器从 draft 加载并感知 draft/待确认/置信度；Word 预览栏（复用可折叠面板）；活树上的 **5 选项批量层级标定**（原地重排、藏存不丢、不静默丢失）；待确认 triage UI；「完成」按钮与校验。
- **P3 · 入口统一与下线**
  统一"新建（空白 / 从 Word）"入口；下线 beta2 导入对话框与老导入路径、清理 `import-v2/*` 只读卡片；灰度开关 + 迁移。

**节奏**：本总览 spec 固化模型；随后 **P1 打头**（地基），逐 Phase 各写自己的 spec 与计划。

## 7. 风险与灰度

- **大重构动稳定流程**：动 `procedureEditorStore`、后端草稿化、docx 存取、层级标定改造、列表/状态——回归面大。→ **灰度开关 + 增量替换**，不大爆炸式上线；新旧路径并存到 P3 才下线旧路。
- **孤儿草稿**：上传即落库会产生废弃 draft。→ draft 状态 + 自动清理策略（具体阈值进 P1 spec）。
- **降型丢数据**：唯一可能丢数据处。→ "藏存不丢" + "不静默丢失"原则兜底。
- **存储增长**：永久存原始 docx。→ 接受（追溯需求优先）；可在 P1 spec 评估配额/清理。

## 8. 留给各 Phase spec 的待定项

- draft 自动清理的触发条件与阈值（P1）。
- 节点上待确认/置信度/降型藏存的**精确 schema**（P1/P2）。
- 5 选项批量视图在编辑器里的**具体交互与布局**（P2，可能用到可视化 mockup）。
- 「完成」校验的**具体规则**（P2）。
- 灰度开关粒度与迁移步骤（P3）。
