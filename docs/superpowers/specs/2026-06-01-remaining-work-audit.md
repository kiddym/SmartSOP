## 1. 重大更正：SOP×工单执行**已实现 ~85%**（推翻早前判断）

早前一次浅查（grep 不全 / 旧基线）误判"`WorkOrderStepResult` 不存在、SOP×工单融合 ~0%"。**实测推翻此结论**——这是项目的差异化核心（需求 #1），后端其实已基本就绪：

| 能力 | 状态 | 证据 |
|---|---|---|
| 工单挂载 SOP（锁定版本） | ✅ | `work_orders` 含 `procedure_id/procedure_group_id/procedure_attached_at`；`POST /work-orders/{id}/attach-procedure`（仅 PUBLISHED 可挂、不可重复挂） |
| 执行行模型 `WorkOrderStepResult` | ✅ | `response`(JSON)/`notes`/`is_done`/`done_by_user_id`/`done_at`；`(work_order_id,node_id)` 唯一；冗余 `node_code/node_sort_order` 使执行视图自包含 |
| 逐步执行 | ✅ | `attach_procedure()` 为所有 step 节点生成执行行；`execution_view()` 返回 outline+steps；`update_step()` 含必填校验 |
| 完成门控 | ✅ | `assert_completable()`：挂 SOP 的工单须所有执行行 done 才能转 COMPLETE |
| 版本锁定不变性 | ✅ | 钉定 procedure 版本，SOP 后续修改不影响在途工单 |
| 测试 | ✅ | `test_work_order_execution_service.py` 279 行 |
| **执行 PDF 导出 / 签名 / 多媒体响应** | ✗ | 执行记录无法导出 PDF、`WorkOrderStepResult` 无 signature 字段、response 无照片上传 —— **最后一公里缺失** |

| **采购单/供应商/客户** | ✅ ~98% | ✗ 0% | 后端完整（审批→入库）；前端全缺 |
| **分析仪表盘** | ~90% | ✗ 0% | 后端 4 仪表板+MTBF/MTTR+CSV；前端**无任何可视化呈现** |
| **Phase 6 商业化**（订阅/套餐/计费/门控） | ✗ 0% | ✗ 0% | 完全未做（Company 有 plan/subscription_status 预留字段，无逻辑） |
| **Phase 7 移动端** | ✗ 0% | ✗ 0% | 未启动（设计上后期） |

**综合估算：**
- **后端**：phase 0–5 ≈ **85% 完成**（含差异化核心 SOP×工单执行）；缺工时成本、执行 PDF/签名、用户生命周期补全、phase 6 全部。
- **前端**：≈ **20%**（仅 SOP 一条垂直线完整；15 个 CMMS 业务模块前端全缺，连登录页都没有）。
- **端到端可交付产品**：≈ **45–50%**。后端把分子拉高了，但**前端是绝对瓶颈**——后端有引擎，没有驾驶舱。

## 3. 真实剩余工作清单

### A. 后端补齐（相对小，分散）
1. 工单**工时/成本**：Labor(计时器/hourlyRate) + TimeCategory + AdditionalCost + 工单总成本汇总（维护域 0%）。
2. 工单执行**最后一公里**：执行记录 PDF 导出（复用 ReportLab）+ 执行签名字段 + 步骤照片证据（关联 Attachment）。
3. 工单补全：首次响应时间(SLA)、反馈、归档状态、附件绑定端点、看板/日历端点。
4. 资产：树停机向上/向下传播、自动停机触发。
5. 用户生命周期：邮箱激活、密码重置（邮件 token）、用户邀请（UserInvitation+邮件）。
6. 杂项：币种字段、低库存自动邮件告警、PM 日历投影端点。

### B. 前端全栈（最大块，关键瓶颈）
> 后端 33 个 router 几乎都就绪，前端是"照着 API 建界面"——但量很大，且要先建认证地基与全局导航壳。
- **P0 认证地基**：登录/注册/密码找回页、token 拦截器、路由守卫、全局导航壳（AppSidebar 现在只有"程序库/草稿箱"两项，需扩成 CMMS 全模块导航）。**所有 CMMS 前端的前置。**
- **P1 工单 + SOP 执行界面**：工单列表/看板/详情 + **逐步执行界面**（对接已就绪的 `execution_view`/`update_step`）。这是差异化核心的可视化，"为什么买"的演示点。
- **P1 资产 + 位置**：树形列表/详情/分类/停机（工单的依赖主数据）。
- **P2 库存采购**：备件/消耗/采购单审批/供应商客户。
- **P2 PM + 计量 + 请求**：计划/排程/读数/触发器/请求审批。
- **P3 分析仪表盘**：对接 4 个分析端点做可视化。
- **P3 组织与平台**：用户/角色/团队/通知中心/公司设置。

### C. Phase 6 商业化（前后端皆 0）
订阅/套餐(FREE/STARTER/PRO/BUSINESS)、座席门控、功能门控、计费 webhook(Stripe)、自托管 license。

### D. Phase 7 移动端
技师现场执行（工单/资产/扫码/离线），后期。

### E. 横切
i18n 落地（前端已有部分 auth 字符串）、parity 矩阵持续勾销对账、批量解析 SOP（本仓已出 3 份 plan，SOP 侧增强）。

## 4. 优先级建议（让产品先"可用"的关键路径）

按"最快得到一个可演示、可卖的闭环"排序：

1. **前端认证地基 + 导航壳**（P0）—— 不做这个，后端再全也没人能用。
2. **前端工单 + SOP 逐步执行界面**（P1）—— 差异化核心的 UI，对接已就绪后端，是 demo 的高光。
3. **前端资产 + 位置**（P1）—— 工单挂载的依赖数据。
4. **后端工时成本 + 执行 PDF/签名**（P1 后端）—— 补全工单执行的合规闭环（与 #2 的 UI 配套）。
5. **前端库存采购 / PM / 计量 / 请求**（P2）。
6. **前端分析仪表盘 + 组织管理（用户/角色/团队/通知）**（P3）。
7. **Phase 6 商业化**（要收费时）。
8. **Phase 7 移动端**（后期）。

> 关键判断：**当前项目的瓶颈不在后端，而在前端**。后端（含最难的 SOP×工单执行）已 ~85%，但前端只有 SOP 一条线。下一阶段的重心应从"后端造功能"转向"前端把已就绪的后端能力变成可用界面"，从认证地基与工单执行界面开刀。

## 5. 与"批量解析三份 plan"的关系

批量解析（已出 Plan 1/2/3）是 **SOP 模块的内部增强**，属上表"SOP 模块"那一行的深化，**不推进 CMMS 主线任何 phase**。它与本盘点的主线工作（前端全栈 / 商业化）是并行的两条线——做不做、何时做，取决于"批量导入历史 SOP"这个需求相对"让 CMMS 可用"的优先级。建议：若近期目标是产品可用，主线前端优先于批量解析。
